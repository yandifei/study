# Redis 知识要点笔记

## 一、关于“Redis 所有值都是字符串”的说法

- **对外表现**：所有键和值在使用命令时都以字符串形式提供。
- **内部实现**：Redis 会针对不同数据类型和内容自动选择高效的编码方式，例如：
  - String 类型：`int`（整数）、`embstr`（短字符串）、`raw`（长字符串，通过 SDS 存储）。
  - 其他类型（List、Hash、Set 等）底层使用 hashtable、ziplist/listpack、skiplist 等结构。
- **结论**：可以说“最终都是字符串”，但内部并非简单的 C 字符串，而是多种优化结构的组合。

## 二、Redis 数据存储安全措施

### 1. 网络与访问控制
- 监听内网，通过防火墙/安全组设置 IP 白名单，**禁止公网直接访问**。
- 设置强密码：`requirepass <password>`。
- 使用 ACL（Redis 6.0+）创建不同权限用户，遵循最小权限原则。
- 禁用或重命名高危命令：`FLUSHALL`、`CONFIG`、`KEYS` 等。
- 保持 `protected-mode yes`，避免未授权访问。

### 2. 数据加密
- **传输加密**：启用 TLS 加密客户端与服务端、集群节点之间的通信。
- **静态数据加密**：开源 Redis 不支持透明加密，可选替代方案：
  - 应用层加密（写入前加密，读取后解密）。
  - 文件系统加密（如 LUKS、云盘加密）。
  - 使用 Redis Enterprise 等商业版本。

### 3. 持久化与高可用
- 持久化：RDB 快照 + AOF 日志，防止进程重启后数据丢失。
- 备份：定期将持久化文件备份到异地。
- 高可用：主从复制、哨兵（Sentinel）、集群（Cluster）。

## 三、Redis 二进制安全与中文支持

### 核心概念
- **二进制安全**：Redis 不依赖特殊分隔符（如 `\0`），而是根据数据长度存储和读取任意字节序列。
- **正因为二进制安全，Redis 才能完美支持中文、图片、序列化对象等任何数据**。

### 中文的存储过程
1. 客户端（如 redis-cli）将中文字符按 UTF-8 编码为字节序列。
2. Redis 以二进制安全方式原样存储这些字节。
3. 读取时 Redis 返回原始字节，客户端再按 UTF-8 解码为中文。

### 实际操作验证
```bash
127.0.0.1:6379> SET 姓名 "张三"
OK
127.0.0.1:6379> GET 姓名
"张三"
```

### redis-cli 显示十六进制转义的原因及解决
- **原因**：redis-cli 默认将非打印字符或可能引起歧义的字节显示为转义形式（如 `\xe5\xa7\x93`），这是安全行为，**不代表存储错误**。
- **解决**：启动时添加 `--raw` 参数，使 redis-cli 直接显示 UTF-8 解码后的字符。
  ```bash
  redis-cli --raw
  127.0.0.1:6379> KEYS *
  姓名
  ```
- 通过编程客户端（Python、Node.js 等）获取后按 UTF-8 解码同样正常。

## 四、常见误解澄清

| 误解                              | 正解                                                    |
| --------------------------------- | ------------------------------------------------------- |
| “二进制安全 = 不支持文本”         | 二进制安全 = 支持任意二进制数据，包括文本（任何语言）。 |
| “redis-cli 显示乱码 = 存储出问题” | 仅是显示转义，存储的数据是完整正确的 UTF-8 字节。       |
| “Redis 内部只有字符串”            | 内部有多种高效编码，只是对外表现为字符串。              |
