针对你追问的两个核心问题，这里把“文档关系图”彻底讲透。

---

### 一、文档关系图是什么？怎么画？
**文档关系图** 是用于可视化 MongoDB 数据模型的图，它不画表与表的关系，而是画**文档内部结构**以及**集合之间的引用关系**。  
画法可以分两层：

#### 1. 集合内：文档结构草图
直接用 **JSON 或注释形式** 标出一个文档包含哪些字段，哪些是嵌入子对象/数组，哪些是字段存的是引用 ID。

**例子：博客系统的用户、文章、评论**
```json
// 集合 users
{
  "_id": ObjectId,
  "username": "alice",
  "email": "a@b.com",
  "profile": {                // 嵌入对象（1:1 少，经常一起读）
    "bio": "string",
    "avatar_url": "url"
  },
  "recent_post_ids": [        // 引用 ID 数组（只存最少的关联信息）
    ObjectId, ObjectId
  ]
}

// 集合 posts
{
  "_id": ObjectId,
  "title": "string",
  "content": "string",
  "author_id": ObjectId,      // 引用 users._id
  "tags": ["mongodb", "python"], // 简单数组
  "comment_ids": [ObjectId, ...] // 引用 comments._id（如果评论很多，也可不存，靠评论集合反查）
}

// 集合 comments
{
  "_id": ObjectId,
  "post_id": ObjectId,        // 引用 posts._id
  "user_id": ObjectId,        // 引用 users._id
  "text": "string",
  "created_at": ISODate
}
```
这种草图就是**最直接的文档关系图**——在团队里白板画或在文档里用类似结构注释即可。

#### 2. 集合间：实体关系图（ER 的变体）
你可以用带箭头的图示表示集合之间的引用方向：
- 矩形代表集合名（如 `users`, `posts`, `comments`）
- 连线标注引用字段及类型（1:1, 1:N, M:N）
- 如果是嵌入，直接在父文档矩形内画出子对象结构。

很多专门工具能将这种图形直接生成代码或 schema。  
**推荐工具**：
- **Moon Modeler**：专门支持 MongoDB 的图形化建模工具，可导出 Schema。  
- **Hackolade**：强大的多数据库建模工具，原生支持 MongoDB 嵌套和引用。  
- **dbdiagram.io**：在线 ER 工具，虽偏向关系型，但也可通过注释表示嵌入/引用。  
- **手绘/白板 + JSON 草图**：对快速沟通最有效。

**官方文档参考**：
- MongoDB 数据建模指南（英文）：[Data Modeling Introduction](https://www.mongodb.com/docs/manual/data-modeling/)
- 嵌入与引用的详细场景判断：[Embedded Data Versus References](https://www.mongodb.com/docs/manual/data-modeling/embedded-data-models/)

### 二、有需要遵循的数据库范式吗？
**不需要严格遵循关系型范式（1NF, 2NF, 3NF），但要遵循文档型数据库自己的设计原则。**

因为 MongoDB 的优化目标不是消除冗余，而是**按查询模式设计，让读/写性能最优**。  
但完全无政府状态也会出问题，你需要遵循以下“MongoDB 范式”：

#### 1. 原则一：一起读的数据放一起（嵌入优先）
如果几个对象总是一起查询、一起修改，且数据量不会无限制增长，就嵌入。  
这直接违背 3NF（第三范式会要求拆成两张表用外键关联），但在 MongoDB 里是正确做法。  
例：用户的多个收货地址直接嵌入用户文档，而不是另建一张地址表。

#### 2. 原则二：独立生命周期、多对多或无限增长用引用
当一个子实体需要独立于父实体被访问，或数量会无限膨胀（如用户订单、聊天消息），就必须拆分集合，通过引用 ID 关联。  
这类似“逻辑外键”，但没有任何数据库层面的约束，完整性靠应用维护。

#### 3. 原则三：避免大文档和无限嵌套
单文档 16MB 是硬上限。即使没到上限，频繁更新的超大文档也会引发性能问题（因为 MongoDB 通常会移动文档）。  
如果一个数组会不断追加且长度不可控（如日志），马上拆分集合。

#### 4. 原则四：数据冗余有时是必要的
为了加速读取，可以把某些查询频繁的数据（如文章标题、作者名）冗余存储到多个集合里，减少 `$lookup`。  
这在关系型数据库里是反范式，在 MongoDB 里是常规优化。代价是更新时要手动同步。

#### 5. 原则五：类范式——单调性原则（One Fact One Place 的变形）
同一信息如果存储在多处，必须想清楚它变化的频率和一致性的要求。  
- 变化极少的（如用户名、地区字典）：冗余几乎没问题。  
- 变化频繁的（如库存数量、点赞数）：尽量只存一处，用引用，否则应用端同步成本高。

**总结**：  
没有 SQL 范式的硬性规矩，但围绕“读写模式”有一套柔性的设计纪律。  
你画的“文档关系图”就是用来推演这些原则的——每次画完问自己：  
- 这个数据要和父文档一起读吗？→ 嵌入  
- 它会独立被查询吗？会无限增长吗？→ 引用  
- 查询时需不需要额外的冗余字段来避免 JOIN？→ 适量冗余  

这样画出来的图，自然就符合 MongoDB 的最佳实践了。
