# MongoDB 副本集 (Replica Set) 核心知识点总结

## 1. 副本集架构与核心概念

副本集是一组协同工作的MongoDB服务，共同维护同一份数据的多个副本，以保障数据的高可用性，最多可包含50个节点 。

* 
**主节点 (Primary)**：负责接收客户端发送的所有写请求，并在其自身的操作日志中记录对数据的任何更改 。


* 
**从节点 (Secondary)**：负责维护主节点的数据副本，保持与主节点的数据一致，可以在选举主节点时参与投票 。


* 
**仲裁节点 (Arbiter)**：不存储数据副本，也不处理客户端的读写请求，仅负责在选举主节点时参与投票 。使用仲裁节点可以用最小的资源实现奇数节点投票，满足选举要求 。


* 
**复制 (Replication)**：分为在添加新从节点时触发的“初始复制”，以及完成初始复制后持续触发的“同步复制” 。


* 
**心跳 (Heartbeat)**：副本集中的所有节点每两秒会向彼此发送一次心跳消息，用于监控节点的健康状态 。


* 
**操作日志 (Oplog)**：存储在 `local` 数据库的固定大小集合 `local.oplog.rs` 中，记录数据的更改操作 。



## 2. 主节点选举机制

选举机制是副本集实现自动故障转移的基础，确保主节点不可用时能快速选出新主节点 。

* **触发条件**：
1. 首次初始化副本集时 。


2. 从节点向主节点发送心跳消息，且主节点在10秒内未响应时（主节点被视为不可用） 。


3. 添加了一个具备选举资格的从节点，并在该节点完成初始复制后 。




* 
**选举过程**：优先级大于0的从节点首先会将自己状态更新为候选者并给自己投票，然后向其他节点发送投票请求 。仲裁节点会将票投给优先级最高的候选者 。当候选者获得超过拥有投票权节点总数一半的票数时，将成为新主节点 。



## 3. 部署与初始化

* **核心配置参数**：
* 
`replication.replSetName`：指定副本集的名称（如 "rs0"） 。


* 
`replication.oplogSizeMB`：设置操作日志的大小上限（如 1024MB） 。


* 
`net.bindIp`：网络监听绑定的IP地址 。




* 
**初始化命令**：在 `mongosh` 中使用 `rs.initiate()` 命令初始化副本集，可在此步骤中配置各节点的 `_id`、`host`、`priority` (优先级) 和 `arbiterOnly` 等信息 。


* **节点状态**：
* 
`STARTUP` / `STARTUP2`：节点启动或正在进行初始复制的状态 。


* 
`PRIMARY` / `SECONDARY` / `ARBITER`：主节点、从节点、仲裁节点的正常工作状态 。


* 
`RECOVERING`：节点正在进行同步复制或故障恢复，不接收读请求 。


* 
`ROLLBACK`：旧主节点重新加入集群时，回滚尚未复制到从节点的数据的操作状态 。





## 4. 客户端连接与读取偏好 (Read Preference)

* **连接方式对比**：
* 
**直接连接 (Direct Connection)**：在URI中配置 `directConnection=true`。仅连接指定节点，不自动发现其他节点，不支持故障转移。适用于管理员进行管理、节点状态调试等场景 。


* 
**副本集连接 (Replica Set Connection)**：在URI中配置 `replicaSet=<副本集名称>`。驱动程序自动发现全部节点，自动路由读写请求，支持自动故障转移。推荐生产环境应用程序使用 。




* **读取偏好配置**（控制读请求路由）：
* 
`primary`（默认）：读请求仅由主节点处理 。


* 
`primaryPreferred`：默认主节点处理，主节点不可用时由从节点处理 。


* 
`secondary`：读请求仅由从节点处理 。


* 
`secondaryPreferred`：默认从节点处理，所有从节点不可用时由主节点处理 。


* 
*注意：MongoDB默认不允许在从节点上直接执行读取操作，如果在从节点直接调试，需要先执行 `rs.secondaryOk()*` 。





## 5. 副本集日常运维与配置修改

* 
**状态查看**：使用 `rs.status()` 查看集群实时状态，使用 `rs.conf()` 查看完整配置信息 。


* **节点管理**：
* 添加仲裁节点：`rs.addArb("host:port")` 。


* 添加从节点：`rs.add({host: "host:port"})` 。


* 删除节点：`rs.remove("host:port")` 。




* 
**修改节点属性**：可以通过提取配置 `cfg = rs.conf()`，修改后应用 `rs.reconfig(cfg)` 来调整节点属性 。


* 
**修改优先级**：`cfg.members[index].priority = num` 。


* 
**隐藏节点 (Hidden Node)**：优先级必须设为0（不参与主节点竞选），对客户端直接连接可见，常用于备份数据。配置为 `hidden: true` 。


* 
**延迟节点 (Delayed Node)**：在隐藏节点基础上增加延迟同步机制，用于防止误操作导致的数据丢失（如数据恢复）。配置为 `secondaryDelaySecs: 秒数` 。





## 6. 访问控制与安全性

* 
**Keyfiles 身份验证**：启用访问控制时，可通过指定 Keyfile 认证集群内各节点之间的通信。Keyfile 内容必须是长度介于6到1024之间的 base64 字符串 。


* **启用流程**：
1. 在 `admin` 数据库中创建拥有 `clusterAdmin` 等权限的管理员用户 。


2. 生成密钥文件（如使用 `openssl rand -base64`）并设置权限（如 `chmod 400`） 。


3. 将密钥文件分发给副本集的所有节点 。


4. 修改各节点的配置文件，增加 `security.keyFile` 参数指向密钥文件路径 。


5. 重启集群节点，此后所有客户端连接和操作均需要通过 `db.auth()` 进行授权 。
