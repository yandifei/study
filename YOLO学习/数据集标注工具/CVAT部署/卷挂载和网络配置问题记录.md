# BUG(windows和docker挂载卷不同步)
在Windows系统上使用移动硬盘部署CVAT的Docker Compose环境时，我遇到了容器启动失败的问题。错误提示集中在挂载vector.toml配置文件时出现“not a directory”，随后traefik容器也因无法创建挂载源路径而报错。根本原因是我将项目从原先的D盘移动到了F盘，而Docker Desktop基于WSL2的后端未能正确映射新的盘符，导致宿主机文件在WSL侧的挂载点/run/desktop/mnt/host/f/下状态异常，甚至该挂载点本身变成了普通文件或完全缺失。

为了解决这一问题，我首先确认了F盘下的配置文件实际存在且为普通文件，并检查docker-compose.yml使用了相对路径，避免绝对路径残留。接着，我尝试通过wsl命令进入Docker Desktop专用的发行版，发现/run/desktop/mnt/host/f挂载点不存在，确认了盘符映射失效。随后我打开Docker Desktop的设置，在Resources → File Sharing中手动添加了F盘，然后执行wsl --shutdown彻底关闭所有WSL实例，并完全退出Docker Desktop后重新启动。重启后，再次进入docker-desktop发行版检查，挂载点恢复正常，Windows F盘的内容已正确映射。最后在项目目录下执行docker compose up -d，所有容器成功启动。

这次经历让我认识到，在Windows + WSL2 + Docker Desktop的环境下，容器挂载Windows文件实际上依赖于Docker Desktop对盘符的动态映射。当项目路径改变或挂载点异常时，首要任务是确认Docker Desktop的文件共享设置中是否包含对应的盘符，并确保WSL侧的挂载点正常。通过重启WSL和Docker Desktop可以快速重置挂载状态，避免盲目修改配置文件。这一通用排查思路——先检查共享驱动器，再验证WSL挂载点，最后重启环境——将成为我日后解决类似挂载问题的标准流程。
