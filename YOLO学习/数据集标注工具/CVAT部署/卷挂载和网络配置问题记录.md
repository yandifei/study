# BUG(windows和docker挂载卷不同步)
我遇到的问题是，在Windows系统上使用移动硬盘运行CVAT的Docker Compose环境时，启动容器失败，报错提示挂载vector.toml配置文件时出现“not a directory”错误，原因是Docker尝试将主机上的文件挂载到容器内，但主机路径被识别为目录或不存在。由于项目之前存放在D盘，现在改为F盘，路径变化导致挂载失败。进一步检查发现，Windows物理路径F:\cvat\components\analytics\vector\vector.toml文件存在，但通过Docker Desktop的WSL2后端访问的路径/run/desktop/mnt/host/f/...下文件状态不一致，说明WSL侧未能正确映射Windows盘符，导致Docker无法识别文件类型。

解决方法分为几个步骤：首先确认主机上的配置文件确实存在且为普通文件，不是目录；其次检查docker-compose.yml中的volumes挂载使用相对路径，避免绝对路径残留；然后通过wsl命令进入Docker Desktop使用的发行版，检查/run/desktop/mnt/host/f/下文件是否同步；接着在Docker Desktop设置中重新共享F盘，并重启Docker Desktop和WSL（执行wsl --shutdown）；最后如果问题依旧，可临时将配置文件复制到WSL内部路径并修改挂载源，绕过跨文件系统挂载问题。

从这次经历可以推论，以后遇到类似的容器挂载错误，应遵循通用排查思路：第一，确认报错中主机路径在宿主机上确实存在，且文件类型与容器期望一致（文件挂载文件，目录挂载目录）。第二，考虑运行环境特性，在Windows + WSL2 + Docker Desktop的组合下，挂载路径实际经过WSL中转，容易因盘符映射、共享驱动器设置、WSL发行版不一致等导致文件识别异常。此时应先检查Docker Desktop的共享驱动器列表，确保所需盘符已启用；再通过wsl进入正确的发行版，验证挂载点内容是否与Windows一致。第三，如果重启Docker和WSL仍无效，可以尝试使用绝对WSL路径或Docker卷来替代直接挂载Windows文件，确保稳定性。理解Docker在WSL2下的文件系统映射原理（Windows路径通过/run/desktop/mnt/host/或/mnt/挂载），能帮助我们快速定位问题根源，避免盲目修改配置文件。
