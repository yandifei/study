## 二、Token 自动刷新 —— 请求拦截器

### 2.1 问题

JWT `access_token` 短效期，中途过期返回 `401`。每个页面都手写「判断 401 → 调 `/auth/refresh` → 存新 token → 重试」会大量重复代码。

### 2.2 方案：`utils/request.js`

封装 `wx.request`，全项目统一入口。核心机制：

```
业务代码调用 request()
       │
       ▼
自动挂 Authorization: Bearer <token>
       │
       ▼
  发起 wx.request
       │
  ┌────┴────┐
 200        401
  │          │
  ▼          ▼
直接返回    进入刷新队列
              │
         ┌───┴───┐
       成功      失败
        │        │
        ▼        ▼
     存新token  清空token
     重试队列   跳转登录页
```

### 2.3 刷新锁（防并发 401）

同一时刻多个请求同时 401 → 只发一次 `/auth/refresh`，其余排队等结果：

```js
let isRefreshing = false;
let refreshQueue = [];

// 401 时：
refreshQueue.push({ options, resolve, reject });
if (!isRefreshing) {
  isRefreshing = true;
  doRefresh().then(retryQueue).catch(failQueue);
}
```

### 2.4 使用方式

```js
const request = require('../../utils/request.js');

// 需认证：自动挂 token + 遇 401 自动刷新
request.get(API.USER_INFO).then(res => { ... });
request.post(API.XXX, { key: 'val' }).then(res => { ... });

// 登录等入口接口：跳过拦截
request({ url: API.SEND_CODE, method: 'POST', data: {...}, skipAuth: true });

// 登录成功后统一存储 token
request.saveTokens(res.data);
```

### 2.5 改造成效

| 文件 | 改前 | 改后 |
|---|---|---|
| `app.js` | 112 行（含 `tryRefreshToken`/`clearTokenAndGoLogin`） | 51 行 |
| 业务页面 | 每次手写 `header: { Authorization }` + `wx.getStorageSync` | 零样板代码 |

## 三、后端接口响应格式变化

### 3.1 格式演进

后端统一返回格式：

```json
// 旧：data 直接是数组
{ "code": 200, "msg": "success", "data": [...] }

// 新（分页）：data 包了一层分页对象
{ "code": 200, "msg": "success", "data": { "limit": 20, "skip": 0, "records": [...] } }
```

### 3.2 前端踩坑

- 写 `res.data.data` 拿数组 → 新格式返回对象 → 解析失败 → 列表为空
- **教训：** 前后端对接时必须确认响应数据结构，不能用前端猜测替代真实数据

### 3.3 正确写法

```js
// 浏览记录
const data = res.data.data.records || [];

// 收藏（暂未分页）
const data = res.data.data || [];
```

### 3.4 时间字段名

后端 Pydantic 模型的 `browse_time` / `favorite_time`，前端 WXML 不能写成 `browsed_at` / `favorited_at`。

## 四、浏览记录分页懒加载

### 4.1 接口参数

```
GET /browse/?limit=20&skip=0   → 首页 20 条
GET /browse/?limit=20&skip=20  → 第二页 20 条
GET /browse/?limit=20&skip=40  → 第三页 20 条
```

### 4.2 前端实现

```js
const PAGE_SIZE = 20;

// 首次加载 / 下拉刷新：skip=0，替换列表
loadList() {
  request.get(API.BROWSE_LIST + `?limit=${PAGE_SIZE}&skip=0`)
    .then(res => {
      const data = fmtList(res.data.data.records);
      this.setData({
        list: data,
        skip: data.length,
        hasMore: data.length >= PAGE_SIZE
      });
    });
}

// 触底加载更多：追加列表
onReachBottom() {
  if (!this.data.hasMore || this.data.loadingMore) return;
  this.setData({ loadingMore: true });
  const skip = this.data.skip;
  request.get(API.BROWSE_LIST + `?limit=${PAGE_SIZE}&skip=${skip}`)
    .then(res => {
      const data = fmtList(res.data.data.records);
      this.setData({
        list: [...this.data.list, ...data],
        skip: skip + data.length,
        hasMore: data.length >= PAGE_SIZE,
        loadingMore: false
      });
    });
}
```

### 4.3 防护锁

- `hasMore`：数据加载完毕不再请求
- `loadingMore`：加载中不重复触发

## 五、ISO 时间截断问题

`2026-06-10T17:41:23.423000` 长 27 字符，在 226rpx 宽的网格格子里必然溢出截断。

### 解决方案

```js
// "2026-06-10T17:41:23.423000" → "06-10 17:41"（8 字符）
function fmtTime(iso) {
  return iso ? iso.slice(5, 16).replace('T', ' ') : '';
}

// 数据预处理时附加 displayTime 字段
function fmtList(records) {
  return (records || []).map(r => ({ ...r, displayTime: fmtTime(r.browse_time) }));
}
```

## 六、wx.previewImage 导致页面无故刷新

### 6.1 问题链路

```
点击图片
  → wx.previewImage 打开
  → 页面 onHide
  → 用户关闭预览
  → 页面 onShow 触发
  → loadList() 重新请求第一页
  → 列表重置，滚动位置丢失
```

### 6.2 解决方案：`_justPreviewed` 标记

```js
// 预览前标记
previewImage(e) {
  this._justPreviewed = true;
  const urls = this.data.list.map(item => item.image_url);
  wx.previewImage({ urls, current: url });
},

// onShow 时跳过
onShow() {
  if (this._justPreviewed) {
    this._justPreviewed = false;
    return;  // 预览返回，不刷新
  }
  this.loadList();
},
```

## 七、scroll-view 的三大坑

### 7.1 坑一：refresher 被误触

`refresher-enabled="true"` 绑定在 scroll-view 上，滚动过程中容易被意外触发 → `onRefresh` 重置列表。

**解法：** 去掉 scroll-view 的 `refresher-enabled`，改用页面级下拉刷新：

```json
// json
{ "enablePullDownRefresh": true }

// js
onPullDownRefresh() {
  // 刷新逻辑...
  wx.stopPullDownRefresh();
}
```

页面级下拉只在**页面顶部**下拉才触发，中间滚动不会误触。

### 7.2 坑二：离屏 DOM 被销毁 —— 图片重新请求

scroll-view 为了性能会销毁滚动出视口的 DOM 节点。滚远了再滚回来 → `<image>` 组件重新创建 → 重新发网络请求 → 图片闪白。

**解法：** 去掉 `<scroll-view>`，改用**页面级自然滚动**。DOM 节点始终保留在页面中，图片缓存复用：

```xml
<!-- 改前 -->
<scroll-view scroll-y="true" class="grid-area">
  <view wx:for="{{list}}">...</view>
</scroll-view>

<!-- 改后：直接放在页面中 -->
<view wx:for="{{list}}">...</view>
```

页面级滚动同时提供：
- `onReachBottom` — 触底加载更多（页面级生命周期，无需 `bindscrolltolower`）
- `onPullDownRefresh` — 下拉刷新（页面级，需 json 开启）
- 底部 padding 留出 tabBar 空间（`padding-bottom: 120rpx`）

### 7.3 坑三：vh + calc 高度计算不可靠

`height: calc(100vh - 160rpx)` 在部分微信版本中行为不一致。

**解法：** 去掉固定高度，让内容自然撑开，配合 `page { overflow-y: auto }`。

## 九、502 Bad Gateway 排查

502 是服务端/代理层错误，非前端问题。排查步骤：

1. **直接 curl 后端** — `curl -X POST http://IP:port/endpoint`，不通说明服务没跑
2. **检查 IP 是否漂了** — DHCP 可能重新分配局域网 IP
3. **检查端口监听** — `netstat -ano | findstr "端口"`
4. **检查防火墙** — Windows 防火墙可能拦截入站
