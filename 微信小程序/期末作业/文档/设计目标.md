## 设计目标与原则

### 一、核心愿景：从“集成”到“融合”

**不再做功能的简单堆砌，而是围绕一个明确主题，让图片浏览与AI对话产生化学反应。**

- **旧思路问题**：图片库是图片库，AI是AI，两者割裂，用户不知道为什么要在一个应用里同时使用两者。
- **新设计目标**：以**图片为媒介，AI为能力**，打造一个“**可看、可聊、可思考**”的ACG智能图廊。用户不仅欣赏图片，还能与AI深度交流图片背后的风格、技法、创作灵感。

**一句话定义**：  
> **一个由AI驱动的、面向ACG爱好者的图片鉴赏与智能对话小程序。**

### 二、明确主题：ACG + 智能对话

**我要做的是一个“有智商的图廊”，而不是又一个壁纸合集。**

- **主题聚焦**：内容以ACG（动画、漫画、游戏）为核心，兼顾PC壁纸、风景、AI绘图等泛视觉艺术。
- **智能特色**：
  - 图片有**AI预生成的描述与标签**（让机器先看懂）。
  - 用户可以**与AI多模态对话**（让用户与机器共赏、追问、发散）。
  - AI不仅能描述，还能**分析风格、解读构图、推荐相似作品**（比单纯标签更深入）。

**服务群体**：  
- **核心用户**：ACG爱好者、画师、图片收藏控、对AI感兴趣的技术艺术交叉人群。
- **使用场景**：碎片化浏览、灵感收集、图片深度解读、AI绘画前参考、纯粹与AI聊天玩图。

### 三、数据与结构：清晰、解耦、可扩展

**明确“存什么、怎么存、谁提供、谁消费”。**

| 数据域                  | 存储位置             | 格式/结构    | 提供方             | 消费方                    |
| ----------------------- | -------------------- | ------------ | ------------------ | ------------------------- |
| 用户信息、收藏、浏览    | MongoDB (users集合)  | 嵌入式子文档 | 数据库服务 (21325) | 前端小程序                |
| 图片元数据（URL、类型） | MongoDB (images集合) | 扁平文档     | 管理员导入脚本     | 数据库服务                |
| AI预生成描述/标签       | 已预置在图片文档？   | 文本/数组    | 离线模型生成       | 数据库服务（随图片返回）  |
| 用户-AI对话内容         | Dify平台             | 会话+消息    | Dify API           | 前端通过AI服务(21326)调用 |
| 会话期间上传的图片      | Dify临时存储         | 文件ID       | 用户实时上传       | Dify多模态模型            |

**关键约束**：  
- 前端不直接调用Dify，必须经过自己的AI服务（加鉴权、用户映射）。  
- 数据库服务只负责“静态图片资源”，AI服务负责“动态对话能力”。  
- 浏览/收藏记录**嵌入用户文档**，避免联表查询，提升读取性能。

### 四、前后端交互：约定明确、可测试、可模拟

**通信不是“能通就行”，而是要有清晰的契约和防呆机制。**

| 交互类别                        | 通信方式                                   | 数据格式                              | 认证/安全                     | 特殊处理                          |
| ------------------------------- | ------------------------------------------ | ------------------------------------- | ----------------------------- | --------------------------------- |
| 常规业务API（登录、收藏、浏览） | HTTPS → Nginx → /db/* → 数据库服务         | JSON + 统一响应结构 `{code,msg,data}` | JWT (Access Token 15min)      | Token过期自动刷新（Rotation）     |
| 图片列表/详情                   | GET /images?topic=&skip=&limit=            | JSON数组                              | 游客可读，但收藏等需登录      | 缩略图URL提前裁剪好               |
| AI流式对话                      | SSE (POST /ai/v1/chat-messages)            | text/event-stream                     | API Key（后端存）+ 用户ID映射 | 支持停止生成、历史管理、反馈      |
| 会话历史                        | REST (GET /ai/v1/conversations, /messages) | JSON                                  | 同上                          | 前端需反转消息顺序（API返回倒序） |

**关键约定**：  
- 所有AI相关的`user`字段，必须通过数据库服务的`GET /auth/me`获取真实用户ID，不能使用前端传入。  
- 小程序请求统一通过`utils/request.js`拦截，实现**自动添加Token、401处理、加载提示、错误吐司**。  
- 开发环境直连后端端口，生产环境走Ngrok HTTPS，但前端代码**只维护一套Base URL配置**（通过环境变量切换）。

## 总结：一句话设计目标

> **以ACG图片为核心内容，以多模态AI对话为核心交互，构建一个数据清晰、前后端解耦、可扩展的“智能图廊”微信小程序——让每一张图都可聊，每一次对话都有图。**

你可以把这个设计目标放到你的需求文档**第1章前面作为“设计哲学”**，也可以单独抽出来作为团队对齐的纲领。它回答了你关心的所有问题：

- **做什么**：ACG智能图廊（图片+AI对话）
- **用什么**：微信小程序 + FastAPI + MongoDB + Redis + Dify
- **服务谁**：ACG爱好者、图片收藏者、AI兴趣用户
- **数据怎么存**：用户相关嵌入user文档，图片独立集合，AI对话由Dify托管
- **前后端怎么聊**：业务API走REST+JWT，AI对话走SSE+独立服务
