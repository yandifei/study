# 页面间传参
微信小程序里实现页面间传参，最好的方法是使用 **EventChannel**。它能像通道一样，在页面跳转的同时，高效、优雅地传递复杂数据。

我会先为你介绍这个最佳方案，然后也会提供几种备选方法，方便你在不同场景下选用。

### ⭐️ 最佳方案：使用 `EventChannel`（官方推荐，适合传递复杂数据）

这是官方提供的首选方案，非常适合你的场景：在第一个页面处理用户点击，然后把数据传给打开的第二个页面当作启动参数。它就像一个专属的“数据通道”，全程由 API 控制，非常可靠。

**具体实现步骤如下：**

**步骤 1: 在第一个页面（A 页面）跳转时发送数据**

当你需要跳转时，在 `wx.navigateTo` 的 `success` 回调中，通过 `res.eventChannel.emit()` 方法将数据发送出去。

```javascript
// pageA.js
Page({
  goToPageB() {
    // 假设这是你要传递的数据，可以是一个复杂对象
    const myData = {
      id: 123,
      name: '示例数据',
      config: { theme: 'dark' }
    };

    wx.navigateTo({
      url: '/pages/pageB/pageB',
      success: (res) => {
        // 通过 eventChannel 向页面 B 发送数据
        // 'sendData' 是一个自定义的事件名
        res.eventChannel.emit('sendData', myData);
      }
    });
  }
});
```

**步骤 2: 在第二个页面（B 页面）加载时接收数据**

第二个页面启动后，在它的 `onLoad` 生命周期里，通过 `this.getOpenerEventChannel()` 获取同一个通道，然后用 `on` 方法监听并接收第一个页面发来的数据。

```javascript
// pageB.js
Page({
  onLoad: function(options) {
    // 获取与上一个页面通信的 EventChannel 对象
    const eventChannel = this.getOpenerEventChannel();
    
    // 监听 A 页面发出的 'sendData' 事件，并接收数据
    eventChannel.on('sendData', (data) => {
      console.log('接收到来自 A 页面的数据:', data);
      // 将接收到的数据存入当前页面的 data 中，以便在 WXML 中使用或作为启动服务的参数
      this.setData({
        serviceParams: data
      });
      
      // 现在，你就可以用 `this.data.serviceParams` 来启动你的服务了
      this.startYourService(this.data.serviceParams);
    });
  },

  startYourService(params) {
    // 这里写你的服务启动逻辑，params 就是从 A 页面传来的参数
    console.log('服务已启动，参数为:', params);
  }
});
```

### 📝 其他备选方案

虽然 `EventChannel` 是最佳实践，但微信小程序也提供了其他几种方法，可以根据你的具体需求来选择。

*   **全局对象 (`getApp().globalData`)**：该方法通过在 `app.js` 中定义 `globalData` 对象，让所有页面都能通过 `getApp()` 来存取数据，实现全局共享。
    *   **优点**：实现简单直接，所有页面都可访问。
    *   **缺点**：数据变化时，依赖该数据的页面无法自动更新，需要手动管理，增加了维护成本。
    *   **适用场景**：适合存放全局配置、用户信息等静态或一次性加载的数据。

*   **本地缓存 (`wx.setStorageSync` / `wx.getStorageSync`)**：该方法利用微信提供的 `Storage` API，将数据持久化存储到用户设备中。
    *   **优点**：数据会持久化保存，即使小程序关闭再打开，数据依然存在，非常适合保存用户偏好设置。
    *   **缺点**：API 调用是异步的（虽然有同步版本），操作相比全局变量稍显繁琐。有容量限制（单个 key 限制 1MB，总体限制 10MB），不适合大数据量或高频率的读写。
    *   **适用场景**：用于需要长期保留的数据，如用户登录态、应用设置项等。

*   **路由 URL 参数**：在跳转时通过 `url` 字符串来传递参数，如 `url: '/pages/pageB/pageB?id=123&name=test'`。目标页面可在 `onLoad` 的 `options` 参数中接收。
    *   **优点**：操作非常简单直接。
    *   **缺点**：URL 长度有限，且只能传递字符串。传递对象需先 `JSON.stringify`，在接收端再 `parse`，略显繁琐且不优雅。
    *   **适用场景**：适合传递少量、简单、类型为字符串的数据，例如“文章ID”、“商品ID”等。

### 📊 方案快速对比

| 方法               | 优点                                                                 | 缺点                                                             | 适用场景                                             |
| :----------------- | :------------------------------------------------------------------- | :--------------------------------------------------------------- | :--------------------------------------------------- |
| **`EventChannel`** | **官方推荐**，专为页面间通信设计，可传复杂对象，无需手动管理全局状态 | 仅在 `wx.navigateTo` 等 API 的 `success` 回调中可用              | **最佳实践**，用于 **A 打开 B** 这种一对一的页面传参 |
| **全局对象**       | 简单方便，实现全局共享                                               | 非响应式，数据更新不会自动触发页面渲染，需谨慎管理以防污染       | 存放**全局配置**、用户信息等静态数据                 |
| **本地缓存**       | 数据持久化，即使小程序关闭再打开依然存在                             | 异步 API，有容量限制，操作相对繁琐                               | 保存需要**长期保留**的数据，如用户登录态、设置项     |
| **URL 参数**       | 实现非常简单直观                                                     | **只能传字符串**，长度有限，无法直接传复杂对象，对象需手动序列化 | 传递简单、少量的数据，如 ID、类型等                  |

# URL 参数（目前的方案）
现在是用 URL 参数（?topic=${code}）来传参，这是最简单直接的方式，对于像 topic 这样的字符串参数完全没问题。
✅ 优点：简单直观，适合单个字符串、数字等简单数据。
❌ 缺点：无法直接传递对象、数组；URL 有长度限制（约 2KB）；特殊字符需要编码。
第一个界面:
设置某个控件与这个事件绑定，当事件触发时它会跳转`gallery界面`
```js
nav(e) {
    const code = e.currentTarget.dataset.url;
    wx.navigateTo({ url: `/pages/gallery/gallery?code=${code}` });
}
```
在目标页面 gallery.js 中，通过 onLoad(options) 接收：
```js
onLoad(options) {
  const topic = options.code;  // 拿到传递的字符串
}
```
