# 小程序“浏览/收藏”功能失效问题排查与解决总结（最终版）

## 1. 问题现象
- 小程序上线（体验版/正式版）后，**浏览记录**和**收藏列表**功能无法加载数据，提示“加载失败”。
- **其他接口**（如登录、获取用户信息、发送验证码等）均可正常访问。
- 在微信开发者工具中开启“不校验合法域名”时功能正常，但在真机或正式环境下报错。

## 2. 排查过程

### 2.1 初步怀疑：接口路径尾部斜杠
- 检查 `config/api.js` 发现浏览/收藏接口均以 `/` 结尾（如 `/db/browse/`），而登录等接口无尾部斜杠。
- 修改为去掉尾部斜杠，问题依旧。

### 2.2 打印错误详情
- 最初 `.catch` 只弹出通用 toast，无法定位错误。
- 修改为 `console.error(err)`，发现 `err` 为空对象 `{}`，`statusCode` 为 `undefined`。
- 说明**请求根本没收到服务器的 HTTP 响应**，直接进入了 `wx.request` 的 `fail` 回调。

### 2.3 进一步输出具体错误
- 再次改造 `.catch`，直接输出错误对象和 `errMsg`。
- 控制台显示：
  ```
  TypeError: Cannot read property 'records' of undefined
  ```
- 说明请求**成功返回了数据**，但数据类型不是预期的 JSON，导致解析失败。

### 2.4 打印真实返回内容
- 在 `loadList` 的 `.then` 中直接 `console.log(res.data)`。
- 输出为一长串 **HTML 页面**，内容为 ngrok 的警告页：
  ```
  ERR_NGROK_6024
  You are about to visit ..., served for free through ngrok.com.
  ```
- 确认请求返回的是 ngrok 的“信任警告”页面，而非后端 JSON。

### 2.5 解决 ngrok 警告页
- 在 `utils/request.js` 中为所有请求统一添加 HTTP 头部 `'ngrok-skip-browser-warning': 'true'`。
- 修复后，请求不再返回 HTML 警告页，而是收到真正的后端响应。

### 2.6 出现 405 Method Not Allowed
- ngrok 问题解决后，浏览/收藏请求开始返回 **HTTP 405**。
- 查看 Network 面板，发现请求路径为 `GET /db/browse?limit=20&skip=0`（无尾部斜杠），而后端 FastAPI 路由定义为 `@router.get("/")`，完整路径为 `/db/browse/`。
- 小程序请求无斜杠时，FastAPI 默认发送 307 重定向至带斜杠的路径。由于 `wx.request` 未正确跟随重定向（或后端关闭了重定向），导致直接返回 405。

### 2.7 修复尾部斜杠
- 在 `config/api.js` 中，将浏览/收藏所有接口路径末尾补上 `/`：
  ```js
  BROWSE_LIST:  base_url + '/db/browse/',
  BROWSE_BATCH: base_url + '/db/browse/',
  BROWSE_CLEAR: base_url + '/db/browse/',

  FAVORITE_LIST:  base_url + '/db/favorite/',
  FAVORITE_ADD:   base_url + '/db/favorite/',
  FAVORITE_CLEAR: base_url + '/db/favorite/',
  FAVORITE_REMOVE: base_url + '/db/favorite/',
  ```
- 修改后请求直接命中 FastAPI 路由，不再触发重定向，405 解决。

### 2.8 验证上线环境
- 本地开发环境（`http://127.0.0.1:61000`）与正式环境（ngrok 域名）均使用相同的路径（带 `/`）和跳过警告头，请求均返回 200，数据正常展示。

## 3. 根本原因总结
本次故障由**两个独立问题叠加**造成：

| 问题               | 原因                                                                     | 影响                                  |
| ------------------ | ------------------------------------------------------------------------ | ------------------------------------- |
| ① ngrok 警告页     | 免费 ngrok 隧道返回的浏览器警告页面，`wx.request` 无法交互跳过           | 请求返回 HTML 而非 JSON，导致解析失败 |
| ② 路径尾部斜杠缺失 | FastAPI 路由定义带 `/`，请求路径不带 `/` 时发生 307 重定向，小程序未跟随 | 后端直接返回 405 Method Not Allowed   |

两者叠加导致前期排查困难：最初去掉斜杠后仍失败（因被 ngrok 警告页掩盖），误以为斜杠不是原因；解决 ngrok 后才暴露出真正的路由匹配问题。

## 4. 最终解决方案

### 4.1 前端修改

**文件：`utils/request.js`**
- 在请求头构造处添加 `'ngrok-skip-browser-warning': 'true'`，避免 ngrok 警告页。

**文件：`config/api.js`**
- 浏览、收藏模块所有接口路径统一以 `/` 结尾，与 FastAPI 路由定义保持一致。

### 4.2 后端无需修改
- FastAPI 默认的斜杠行为符合 RESTful 规范，保持原样即可。

## 5. 经验教训
1. **错误处理必须输出完整信息**：不可仅用通用 toast 掩盖错误，应打印 `err` 对象、`errMsg` 及响应体内容。
2. **遇到非预期格式响应时，先检查原始返回内容**：HTML 页面、重定向、甚至反向代理页面都可能伪装成成功响应。
3. **注意反向代理/内网穿透工具的特性**：免费 ngrok 的警告页可通过 `ngrok-skip-browser-warning` 请求头规避，其他工具也可能有类似限制。
4. **前后端路径约定必须精确匹配**：FastAPI 默认斜杠重定向在小程序环境不可靠，路径中应保留后缀 `/`。
5. **环境差异会导致问题复现不一致**：开发者工具可绕过域名校验和证书校验，真机调试与正式版才是最终验证标准。
6. **长期建议**：免费 ngrok 存在稳定性与限制问题，可考虑替换为 Cloudflare Tunnel、frp 或部署到正式服务器，从根本上减少此类干扰。
