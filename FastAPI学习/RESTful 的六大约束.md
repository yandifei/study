# RESTful 的六大约束（复习笔记）

> 💡 **导读**：RESTful 架构的核心就在于这六大约束。满足这些约束的 Web API 才能被称为真正意义上的 RESTful API。以下是详细的知识点梳理，方便随时复习和背诵。

---

### 1. Client-Server (客户端-服务器分离)

* **核心概念**：前后端各干各的活。
* **具体表现**：前端（客户端）负责数据的**展示和用户交互**，后端（服务器）负责数据的**处理和存储**。
* **主要优势**：两端解耦，互不干扰。前端改版不影响后端逻辑，后端扩容不影响前端展示。

### 2. Stateless (无状态)

* **核心概念**：每次请求都是独立的，服务器不保存客户端的会话（Session）信息。
* **具体表现**：客户端每次向服务器发送请求时，必须在请求中包含**所有必要的信息**（例如身份令牌 Token）。服务器就像一个有“金鱼脑”的店员，不记得你上一秒说了什么，你想干嘛必须在这一次请求里说清楚。
* **主要优势**：服务器不需要消耗内存去记住每个用户的状态，极大地提高了服务器的**横向扩展能力**和高并发处理能力。

### 3. Cacheable (可缓存)

* **核心概念**：服务器的响应可以被标记为“可缓存”或“不可缓存”。
* **具体表现**：对于一些不经常变动的数据（如商品分类、公共配置），服务器会告诉客户端：“这东西一段时间内不会变，你先存着”。客户端下次需要时，直接重用本地缓存数据，不用再跑去问服务器。
* **主要优势**：大幅减少服务器的请求压力，显著提升客户端的加载速度和性能。

### 4. Layered System (分层系统)

* **核心概念**：客户端不需要、也无法知道自己直接连的是真正的服务器还是中间层。
* **具体表现**：在客户端和核心服务器之间，系统可以非常灵活地加入**反向代理、网关、负载均衡器、安全防火墙**等中间层。客户端发出的请求链路对它自身是透明的。
* **主要优势**：极大地提高了系统的**安全性、灵活性和可扩展性**（例如可以通过负载均衡把请求分发到不同的服务器群）。

### 5. Uniform Interface (统一接口)

* **核心概念**：所有资源都通过统一、规范的接口进行访问。
* **具体表现**：
* **资源识别**：每个资源都有唯一的 URI（网址）。
* **统一动作**：使用标准的 HTTP 方法（GET, POST, PUT, DELETE）来表达操作意图。


* **主要优势**：降低了开发者的**理解和学习成本**，提高了接口的**可维护性**和通用性。这是 RESTful 架构最核心的灵魂。

### 6. Code-On-Demand (按需代码) —— *[可选约束]*

* **核心概念**：服务器可以返回可执行的代码给客户端执行。
* **具体表现**：服务器不仅能返回纯数据（如 JSON），还可以直接返回一段脚本代码（比如 **JavaScript** 或者是编译好的 Applet 插件），让客户端下载并在本地直接运行。
* **注意事项**：这是六大约束中**唯一一个可选（Optional）**的约束，在实际的现代前后端分离工作中**很少使用**。
