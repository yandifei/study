### 2.2 各阶段详述

#### 阶段一：需求分析

**产出物**：《需求规格说明书》

**主要工作**：
- 梳理邮件发送系统的业务场景
- 定义 11 种邮件模板类型及其变量
- 明确功能性需求（批量、定时、校验、UI）
- 定义非功能性需求（性能、安全、可用性）
- 制定验收标准

**关键决策**：
- 邮件模板覆盖事务类、营销类、生命周期类三大场景
- 定时发送粒度：分钟级（60 秒扫描间隔）
- 邮箱校验采用 RFC 5322 标准

#### 阶段二：技术设计

**产出物**：《技术设计文档》

**主要工作**：
- 确定分层架构（表示层 → 应用层 → 业务层 → 数据层）
- 技术选型（FastAPI + SQLite + 原生 JS）
- 数据库表结构设计（email_tasks 表 8 个字段）
- RESTful API 接口设计（8 个端点）
- 前端交互流程设计（动态表单、校验面板、主题切换）
- 调度器架构设计（后台 daemon 线程 + 线程安全）

**关键设计决策**：
- 选择 SQLite 而非 MySQL/MongoDB：单机部署，零运维
- 放弃 Jinja2 模板引擎：邮件模板简单，字符串替换足够
- 不使用前端框架：避免构建工具链，降低复杂度
- SMTP 失败必须返回 False：确保调度器正确标记任务状态

#### 阶段三：编码实现

**产出物**：全部源代码

**实现顺序**：
1. **模板层**：11 个 HTML 邮件模板（紫色主题）
2. **发送层**：11 个 `send_*` 函数 + 公共 `_build_and_send`
3. **数据库层**：SQLite CRUD + 11 套模板变量定义
4. **调度层**：`DISPATCH_MAP` 分发 + 后台定时线程
5. **API 层**：8 个 REST 端点
6. **UI 层**：单页 Web 管理界面

## 3. 系统实现

### 3.1 技术架构亮点

**1. 模板-函数映射机制**

通过 `DISPATCH_MAP` 字典实现模板代码到发送函数的动态分发：

```python
DISPATCH_MAP = {
    "VERIFY_CODE": {"fn": send_verification_email, "arg_map": ["email", "verification_code"]},
    "PASSWORD_RESET": {"fn": send_password_reset_email, "arg_map": ["email", "user_name", ...]},
    # ... 共 11 条
}
```

新增模板只需：① 添加 HTML 模板 → ② 编写 send 函数 → ③ 注册到 DISPATCH_MAP → ④ 在 TEMPLATE_DEFINITIONS 中声明字段。

**2. 双阶段邮箱校验**

- **前端预检**：用户手动点击「校验邮箱」，调用后端 Pydantic 逐条验证，显示详细结果面板
- **后端强校验**：创建任务 API 强制使用 `EmailStr._validate()`，杜绝非法数据入库

**3. CSS 变量主题系统**

```css
html[data-theme="dark"] { --bg-card: #231d42; --text-primary: #e8e4f0; }
html[data-theme="light"] { --bg-card: #ffffff; --text-primary: #1e1838; }
```

一套 CSS，两套配色，切换无闪烁，偏好持久化。

### 3.2 v1.1 功能增强

| 功能 | 说明 |
|------|------|

### 4.3 SMTP 失败处理测试

| 编号 | 测试用例 | 预期 | 结果 |
|------|----------|------|------|
| T11 | 假 SMTP 凭据发送 | 任务状态标记为 failed | ✅ |
| T12 | 真 SMTP 凭据发送 | 任务状态标记为 sent | ✅ |
| T13 | 网络不通 | 日志记录 warning，返回 False | ✅ |

**关键修复记录**：

> **Bug #1**：`_build_and_send` 中 SMTP 异常被 `except` 块吞掉，不返回 False，导致调度器误将失败任务标记为 sent。  
> **修复**：所有 `except` 块添加 `return False`，正常路径返回 `True`。  
> **Bug #2**：`dispatch_email` 中 `fn(*args)` 返回值未被检查。  
> **修复**：检查返回值的 bool 类型后再返回。

## 5. 项目总结与展望

### 5.1 项目成果

本项目成功实现了一个完整的邮件批量调度系统，核心成果包括：

1. **11 种邮件渲染模板**，覆盖验证码、密码、订单、物流、促销、资讯等多种场景
2. **批量任务管理**，支持多收件人一次性创建、定时发送、手动触发
3. **基于 Pydantic EmailStr 的邮箱校验**，逐条精准定位错误
4. **黑白主题 Web 管理界面**，支持 TXT 导入和动态表单
5. **完善的项目文档**，包括需求规格说明书、技术设计文档和项目报告书

### 5.2 技术收获

| 方面 | 收获 |
|------|------|
| 架构设计 | 分层架构 + 模板方法模式的实际应用 |
| 异常处理 | SMTP 失败链路必须从头到尾传递状态（非抛异常） |
| 线程安全 | SQLite WAL 模式 + `threading.Lock` 保护写入 |
| 前端 | CSS 变量主题系统，零框架实现 SPA 交互 |
| 工程化 | uv 包管理，依赖精简（12 → 7） |

### 5.3 改进方向

| 优先级 | 改进项 | 说明 |
|--------|--------|------|
| P1 | 批量发送并发化 | 当前是串行逐封发送，可引入 `ThreadPoolExecutor` 并发 |
| P1 | 发送重试机制 | 失败任务可自动重试 N 次（指数退避） |
| P2 | 邮件打开追踪 | 嵌入追踪像素，统计打开率 |
| P2 | 用户认证 | 接入 JWT 登录，区分不同运营人员 |
| P2 | 异步 SMTP | 使用 `aiosmtplib` 替代同步 `smtplib` |
| P3 | Docker 部署 | 提供 Dockerfile 和 docker-compose |
| P3 | Grafana 监控 | 接入 Prometheus 指标导出，可视化发送成功率 |

### 5.4 经验教训

1. **SMTP 错误处理至关重要**：邮件发送的成败判定链路必须完整，中间任何一个环节吞掉异常都会导致状态误判
2. **SQLite 适合单机场景**：本项目数据量小、无分布式需求，SQLite 的零配置优势远超 MySQL
3. **CSS 变量是主题切换的最佳方案**：比两套 CSS 文件或 JS 动态改样式更简洁、性能更好
4. **瀑布模型适合需求明确的小项目**：11 种模板场景在需求阶段就已确定，设计→编码→测试的线性流程效率很高
