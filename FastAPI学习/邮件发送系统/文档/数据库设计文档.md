## 1. 数据库选型分析

### 1.1 选型决策矩阵

| 维度 | SQLite | MySQL | PostgreSQL | MongoDB | Redis |
|------|--------|-------|------------|---------|-------|
| **部署复杂度** | ★★★★★ 零配置 | ★★★ 需服务进程 | ★★★ 需服务进程 | ★★★ 需服务进程 | ★★★ 需服务进程 |
| **运维成本** | ★★★★★ 无需 DBA | ★★ 需日常维护 | ★★ 需日常维护 | ★★ 需日常维护 | ★★ 需日常维护 |
| **单机性能** | ★★★★ 优秀 | ★★★★ 优秀 | ★★★★ 优秀 | ★★★ 良好 | ★★★★★ 极佳 |
| **并发写入** | ★★ 单写者 | ★★★★★ 优秀 | ★★★★★ 优秀 | ★★★★ 良好 | ★★★★★ 优秀 |
| **数据持久性** | ★★★★★ 完整 ACID | ★★★★★ 完整 ACID | ★★★★★ 完整 ACID | ★★★ 最终一致 | ★★ 依赖 RDB/AOF |
| **查询语言** | SQL (标准) | SQL (丰富) | SQL (最丰富) | MQL (文档) | 简单命令 |
| **本项目适配度** | ★★★★★ | ★★★ | ★★★ | ★★ | ★★ |

### 1.2 最终选择：SQLite 3

**选择 SQLite 的核心理由**：

1. **本项目的并发特征**：邮件调度系统是典型的 **单进程 + 后台线程** 架构。所有数据库操作发生在同一 FastAPI 进程内，没有多实例同时写入的场景。SQLite 在单机单进程下的性能完全满足需求。

2. **数据量级匹配**：邮件任务数据日均增量预计 < 1000 条，SQLite 在百万级数据量下仍能保持毫秒级响应。

3. **部署零依赖**：不需要安装和配置额外的数据库服务，`uv run main.py` 即可启动整个系统。

4. **数据可移植性**：`email_tasks.db` 是单个文件，备份只需 `cp`，迁移只需 `scp`。

5. **ACID 事务保障**：SQLite 支持完整的原子性、一致性、隔离性、持久性事务，配合 WAL 模式解决了读写并发冲突。

### 1.3 不选其他数据库的原因

| 数据库 | 不选原因 |
|--------|----------|
| **MySQL** | 需要独立服务进程，本项目数据规模和并发量完全不需要 C/S 架构数据库，引入反而增加部署步骤和故障点 |
| **PostgreSQL** | 功能过剩（地理空间、全文搜索等不需要），运维负担过重 |
| **MongoDB** | 本项目数据是严格的表格结构（非文档嵌套），关系型更适合；且原项目已移除 beanie/motor 依赖 |
| **Redis** | 纯内存存储，不适合作为任务持久化的主存储（虽然可用作缓存层，但当前不需要） |

## 2. 数据库概念模型

### 2.1 实体-关系图 (E-R Diagram)

```
┌─────────────────────────────────────────────────────────────┐
│                        系统边界                                │
│                                                               │
│  ┌──────────┐          ┌──────────────────┐                  │
│  │ 运营人员   │ ──创建──→│   email_tasks     │                  │
│  │ (User)   │          │   (邮件任务)       │                  │
│  └──────────┘          ├──────────────────┤                  │
│         │              │ PK  id           │                  │
│         │              │     user_email   │──→ 收件人邮箱     │
│         │              │     send_time    │──→ 计划发送时间   │
│         │              │     template_code│──→ 引用模板类型   │
│         │              │     template_vars│──→ 模板变量(JSON) │
│         │              │     status       │──→ 任务状态       │
│         │              │     created_at   │──→ 创建时间       │
│         │              │     sent_at      │──→ 实际发送时间   │
│         │              └──────────────────┘                  │
│         │                                                     │
│         │  查询/删除/统计                                      │
│         └─────────────────────────────────────────────────────│
│                                                               │
│  ┌──────────────────┐                                        │
│  │  TEMPLATE_        │  (内存中的逻辑实体，非持久化表)           │
│  │  DEFINITIONS      │                                        │
│  │  (11 种模板定义)   │←── template_code 作为外键引用           │
│  └──────────────────┘                                        │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 实体说明

**唯一实体**：`email_tasks`（邮件任务）

> 本系统是单实体模型。`TEMPLATE_DEFINITIONS` 作为模板元数据，定义在 Python 代码中（`utils/email_task_db.py`），不存入数据库。这样设计是因为：
> - 模板定义在开发阶段确定，运行时不频繁变更
> - 模板字段结构复杂（嵌套 JSON），在代码中用 Pydantic 校验比 SQL 约束更灵活
> - 避免了多表 JOIN 查询，简化 SQL

### 2.3 概念模型的关键约束

| 约束 | 描述 |
|------|------|
| 实体完整性 | `id` 为主键，自增唯一 |
| 域完整性 | `status` 仅允许 `pending`、`sent`、`failed` 三值（应用层 CHECK） |
| 引用完整性 | `template_code` 必须存在于 `TEMPLATE_DEFINITIONS` 字典的键中（应用层校验） |
| 非空约束 | `user_email`、`send_time`、`template_code`、`status`、`created_at` 不允许 NULL |
| 默认值 | `template_vars` 默认 `'{}'`，`status` 默认 `'pending'` |

## 3. 数据库关系模型

### 3.1 关系模式

```
email_tasks (━━━━━━━━━━━━━━━━━━━━━━━━━━━━)
  id            : INTEGER   ← 主键，自增
  user_email    : TEXT      ← NOT NULL
  send_time     : TEXT      ← NOT NULL (ISO 8601 格式)
  template_code : TEXT      ← NOT NULL (外键 → TEMPLATE_DEFINITIONS 字典键)
  template_vars : TEXT      ← NOT NULL, DEFAULT '{}' (JSON 字符串)
  status        : TEXT      ← NOT NULL, DEFAULT 'pending'
  created_at    : TEXT      ← NOT NULL (ISO 8601 格式)
  sent_at       : TEXT      ← 可空 (发送成功/失败时才填充)
```

### 3.2 函数依赖

```
id → user_email, send_time, template_code, template_vars, status, created_at, sent_at

说明：
- id 是唯一候选键，决定所有非主属性
- 该关系满足 BCNF（Boyce-Codd 范式）
- 不存在部分依赖、传递依赖
```

### 3.3 范式分析

| 范式 | 是否满足 | 说明 |
|------|----------|------|
| **1NF** | ✅ | 所有字段均为原子值，`template_vars` 虽存储 JSON 但其内部不再拆分查询 |
| **2NF** | ✅ | 单字段主键，不存在部分依赖 |
| **3NF** | ✅ | 不存在传递依赖（如 `send_time` 不决定 `status`） |
| **BCNF** | ✅ | 唯一候选键 `id` 是每个函数依赖的决定因素 |

> **关于 `template_vars` 的 JSON 存储**：将模板变量序列化为 JSON 字符串存储，这属于「有意为之的非规范化」。因为不同模板的变量结构差异巨大（验证码只有 1 个字段，订单确认有 7 个字段含嵌套列表），若按规范化设计需要 11 张子表或 EAV 模式，复杂度远超收益。JSON 列在 SQLite 中可通过 `json_extract()` 查询（虽然本项目未使用此特性）。

### 3.4 关系代数

**查询到期任务**：

```
π[id, user_email, template_code, template_vars]
  (σ[status='pending' ∧ send_time ≤ NOW]
    (email_tasks))
   ORDER BY send_time ASC
```

**统计各状态数量**：

```
status γ COUNT(*)
  (email_tasks)
```

## 4. 数据表详细设计

### 4.1 DDL（数据定义语言）

```sql
CREATE TABLE IF NOT EXISTS email_tasks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_email      TEXT    NOT NULL,
    send_time       TEXT    NOT NULL,
    template_code   TEXT    NOT NULL,
    template_vars   TEXT    NOT NULL DEFAULT '{}',
    status          TEXT    NOT NULL DEFAULT 'pending',
    created_at      TEXT    NOT NULL,
    sent_at         TEXT
);
```

### 4.2 字段详细设计

#### id — 主键

| 属性 | 值 |
|------|-----|
| 数据类型 | `INTEGER` |
| 约束 | `PRIMARY KEY AUTOINCREMENT` |
| 取值范围 | 1 ~ 9,223,372,036,854,775,807 |
| 生成方式 | 数据库自动递增 |
| 用途 | 唯一标识一条邮件任务，API 中用于查询/删除/发送指定任务 |
| 示例 | `1`, `42`, `1024` |

#### user_email — 收件人邮箱

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL` |
| 格式要求 | 符合 RFC 5322 邮箱格式 |
| 校验方式 | 入库前由 Pydantic `EmailStr._validate()` 校验 |
| 示例 | `user@qq.com`, `admin@example.com` |
| 设计考量 | 选择 TEXT 而非单独建用户表 — 本系统不管理用户账户，仅存储收件地址 |

#### send_time — 计划发送时间

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL` |
| 格式 | ISO 8601 标准，如 `2025-06-09T15:30:00` |
| 时区 | 本地时间（与 `datetime.now().isoformat()` 一致） |
| 调度查询 | `WHERE status='pending' AND send_time <= datetime('now')` |
| 即时发送 | 值为当前时间，调度器会立即拾取 |
| 定时发送 | 值为未来时间，到期后调度器自动处理 |

> **为何用 TEXT 而非 DATETIME 类型**：SQLite 没有原生 DATETIME 类型。TEXT 存储 ISO 8601 字符串既保证可读性，又支持字符串比较（ISO 8601 的字典序 = 时间序）。

#### template_code — 模板代码

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL` |
| 取值范围 | `TEMPLATE_DEFINITIONS` 字典的 11 个键 |
| 外键关系 | 逻辑外键 → Python `TEMPLATE_DEFINITIONS` 字典（非数据库级外键） |
| 有效值 | `VERIFY_CODE` / `PASSWORD_RESET` / `PASSWORD_CHANGED` / `LOGIN_ALERT` / `EMAIL_CHANGE` / `ORDER_CONFIRM` / `SHIP_NOTIFY` / `REFUND_STATUS` / `ACCOUNT_WELCOME` / `PROMO_BATCH` / `NEWSLETTER` |

> **设计决策：逻辑外键 vs 数据库外键**：模板定义存储在 Python 代码中（`TEMPLATE_DEFINITIONS` 字典），便于 API 动态返回字段信息。若建 `templates` 物理表，需要在数据库和代码间同步模板元数据，增加维护成本。模板变更频率极低（开发阶段确定），逻辑外键是合理的权衡。

#### template_vars — 模板变量

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL DEFAULT '{}'` |
| 存储格式 | JSON 字符串（`json.dumps(dict, ensure_ascii=False)`） |
| 内容 | 键值对，键为变量名，值为对应的填充内容 |
| 示例 | `{"verification_code": "123456"}` |
| 复杂示例 | `{"user_name":"张三","order_id":"ORD001","items":[{"name":"耳机","price":"199","quantity":"1"}],"total":"199.00"}` |

**各模板的变量结构**：

| 模板代码 | 变量数量 | 复合变量（JSON 数组） |
|----------|----------|----------------------|
| `VERIFY_CODE` | 1 | 无 |
| `PASSWORD_RESET` | 5 | 无 |
| `PASSWORD_CHANGED` | 4 | 无 |
| `LOGIN_ALERT` | 6 | 无 |
| `EMAIL_CHANGE` | 4 | 无 |
| `ORDER_CONFIRM` | 7 | `items[]` |
| `SHIP_NOTIFY` | 6 | 无 |
| `REFUND_STATUS` | 5 | 无 |
| `ACCOUNT_WELCOME` | 3 | `onboarding_steps[]` |
| `PROMO_BATCH` | 8 | `goods[]` |
| `NEWSLETTER` | 4 | `articles[]` |

> **为何用 JSON 字符串而非列拆分**：
> 1. 11 种模板的变量结构完全不同，若拆分为列需要大量 NULL 字段
> 2. 实体-属性-值 (EAV) 模式查询复杂，不符合关系型最佳实践
> 3. JSON 在应用层处理灵活：新增模板只需修改 `TEMPLATE_DEFINITIONS`，无需 ALTER TABLE

#### status — 任务状态

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL DEFAULT 'pending'` |
| 取值范围 | `pending` / `sent` / `failed` |
| 默认值 | `pending` |

**状态转换图**：

```
        ┌─────────┐
        │ pending │ ← 初始状态（创建任务时）
        └────┬────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
┌──────┐         ┌────────┐
│ sent │         │ failed │
└──────┘         └────────┘
 发送成功          发送失败
 (SMTP 返回 OK)   (SMTP 异常/网络错误)
```

**状态迁移规则**：
- `pending → sent`：仅在 `_build_and_send()` 返回 `True` 时
- `pending → failed`：`_build_and_send()` 返回 `False` 或调度器捕获异常
- `sent` 和 `failed` 均为终态，不可再迁移

#### created_at — 创建时间

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | `NOT NULL` |
| 格式 | ISO 8601，`datetime.now().isoformat()` |
| 生成方式 | 服务端在 INSERT 时自动填充 |
| 示例 | `2025-06-09T15:37:26.123456` |

#### sent_at — 实际发送时间

| 属性 | 值 |
|------|-----|
| 数据类型 | `TEXT` |
| 约束 | 可空 (`NULL`) |
| 格式 | ISO 8601 |
| 填充时机 | 状态从 `pending` 变为 `sent` 或 `failed` 时 |
| NULL 含义 | 任务尚未被调度器处理 |

## 6. SQLite 优势分析

### 6.1 与主流数据库对比

#### vs MySQL

| 对比维度 | SQLite | MySQL | 本项目评估 |
|----------|--------|-------|-----------|
| 安装配置 | 内置 Python 标准库，零安装 | 需安装 MySQL Server + 配置用户/密码/端口 | SQLite 胜 |
| 资源占用 | ~1MB 内存，单文件存储 | ~300MB+ 内存，多文件存储 | SQLite 胜 |
| 并发模型 | 单写者（WAL 模式下读写可并发） | 多线程/多进程并发读写 | 本项目够用 |
| 备份恢复 | 复制 `.db` 文件即可 | 需 `mysqldump` 或物理备份 | SQLite 胜 |
| 网络访问 | 不支持（嵌入式） | 支持 | 本项目不需要 |
| 最大数据量 | 140 TB (理论) | 理论上无上限 | 均满足 |

#### vs MongoDB

| 对比维度 | SQLite | MongoDB | 本项目评估 |
|----------|--------|---------|-----------|
| 数据模型 | 关系型（表格） | 文档型（BSON） | 本项目数据是严格表格结构 → SQLite 胜 |
| 查询语言 | SQL（通用标准） | MQL（专有语法） | SQL 更通用 |
| 事务支持 | 完整 ACID | 4.0+ 支持多文档事务 | SQLite 胜（更成熟） |
| Schema 灵活性 | 严格（需 ALTER TABLE） | 动态（无 Schema） | 本项目 Schema 固定 → SQLite 够用 |
| 依赖数量 | 0（Python 内置 `sqlite3`） | 2（`motor` + `pymongo`） | SQLite 胜 |

### 6.2 SQLite 关键特性在本项目中的应用

| SQLite 特性 | 本项目应用 | 收益 |
|-------------|-----------|------|
| **WAL 模式** (`journal_mode=WAL`) | 调度器读 + API 写可并发执行，互不阻塞 | 后台线程读到期任务时，Web UI 仍可创建新任务 |
| **`row_factory = sqlite3.Row`** | 查询结果以字典形式返回 | 代码中 `row["user_email"]` 比 `row[1]` 更可读 |
| **参数化查询** (`?`) | 所有 SQL 使用 `?` 占位符 | 防止 SQL 注入 |
| **自动创建** | 连接时若文件不存在，自动创建 | 首次启动无需手动建库 |
| **单文件存储** | `user_data/email_tasks.db` 一个文件 | 备份只需 `cp`，迁移只需 `scp` |

### 6.3 WAL 模式详解

```
传统 Rollback Journal 模式：       WAL 模式：
┌──────────┐                       ┌──────────┐
│  Writer  │──阻塞──→ Reader       │  Writer  │──写入──→ WAL 文件
└──────────┘                       └──────────┘
                                          │
                                   ┌──────┴──────┐
                                   │   Reader    │──读取──→ 原始 DB + WAL
                                   └─────────────┘
                                   (读不阻塞写，写不阻塞读)
```

**WAL 优势**：
- 读写不互斥：调度器后台查询到期任务时，API 可同时插入新任务
- 写入更快：只需追加到 WAL 文件，不需原地修改数据页
- 崩溃恢复：WAL 文件记录了所有未提交事务，重启后可回滚或重放

**启用方式**：`conn.execute("PRAGMA journal_mode=WAL")` — 每次获取连接时自动设置。

## 7. 并发与线程安全设计

### 7.1 并发场景分析

```
┌─────────────────────────────────────────────────┐
│                   FastAPI 进程                     │
│                                                   │
│  ┌──────────────────────┐  ┌───────────────────┐ │
│  │  主线程 (asyncio)     │  │  后台线程 (Thread)  │ │
│  │                      │  │                    │ │
│  │  • POST /api/        │  │  • scheduler_loop  │ │
│  │    email-tasks (写)  │  │    (读 + 写)       │ │
│  │  • DELETE /api/      │  │                    │ │
│  │    email-tasks (写)  │  │  每 60s:           │ │
│  │  • GET /api/         │  │    读到期任务       │ │
│  │    email-tasks (读)  │  │    更新状态         │ │
│  └──────────────────────┘  └───────────────────┘ │
│                                                   │
│            ┌──────────────┐                       │
│            │  _db_lock     │ ← threading.Lock()   │
│            │  (互斥锁)     │                       │
│            └──────┬───────┘                       │
│                   ▼                               │
│            ┌──────────────┐                       │
│            │ email_tasks  │                       │
│            │    .db       │                       │
│            │ (WAL 模式)   │                       │
│            └──────────────┘                       │
└─────────────────────────────────────────────────┘
```

### 7.2 线程安全措施

| 措施 | 说明 |
|------|------|
| **`threading.Lock()`** | 所有写操作（INSERT / UPDATE / DELETE）在执行前获取 `_db_lock`，确保同一时刻只有一个线程写入 |
| **WAL 模式** | 读操作不获取锁，不阻塞写操作，避免"读阻塞写、写阻塞读"的经典 SQLite 问题 |
| **每次操作新建连接** | `_get_connection()` 每次调用打开新连接，操作完立即关闭，避免连接状态跨线程污染 |
| **`check_same_thread=False`** | 未设置此参数（使用默认 `True`），确保每个线程使用自己的连接对象 |

### 7.3 锁策略

```
读操作 (SELECT, COUNT)：
  不获取 _db_lock → 多个读操作可并发执行

写操作 (INSERT, UPDATE, DELETE)：
  with _db_lock: → 串行化写入
      conn = _get_connection()
      ...执行 SQL...
      conn.commit()
      conn.close()
```

## 8. CRUD 操作规格

### 8.1 操作清单

| 操作 | 函数 | SQL | 锁 | 返回值 |
|------|------|-----|-----|--------|
| **初始化建表** | `init_db()` | `CREATE TABLE IF NOT EXISTS` | 写锁 | — |
| **单条插入** | `insert_task()` | `INSERT INTO email_tasks VALUES (?,?,?,?,?,?)` | 写锁 | `lastrowid` |
| **批量插入** | `insert_tasks_batch()` | N 次 `INSERT` + 1 次 `commit` | 写锁 | `[id1, id2, ...]` |
| **查询到期** | `get_due_tasks()` | `SELECT * WHERE status='pending' AND send_time <= ?` | 无锁(读) | `[{...}, ...]` |
| **查询全部** | `get_all_tasks()` | `SELECT * ... ORDER BY created_at DESC` | 无锁(读) | `[{...}, ...]` |
| **按 ID 查询** | `get_task_by_id()` | `SELECT * WHERE id = ?` | 无锁(读) | `{...}` or `None` |
| **标记已发送** | `mark_task_sent()` | `UPDATE SET status='sent', sent_at=? WHERE id=?` | 写锁 | — |
| **标记失败** | `mark_task_failed()` | `UPDATE SET status='failed', sent_at=? WHERE id=?` | 写锁 | — |
| **删除任务** | `delete_task()` | `DELETE WHERE id = ?` | 写锁 | `bool` |
| **统计计数** | `get_task_count()` | `SELECT COUNT(*) ... WHERE status = ?` | 无锁(读) | `int` |

### 8.2 参数化查询示例

```python
# ✅ 正确：使用 ? 占位符，防止 SQL 注入
conn.execute(
    "SELECT * FROM email_tasks WHERE status = ? AND send_time <= ?",
    ("pending", now_iso)
)

# ❌ 错误：字符串拼接，存在注入风险
conn.execute(
    f"SELECT * FROM email_tasks WHERE status = '{status}'"
)
```

## 9. 数据生命周期

### 9.1 数据流

```
[Web UI 创建任务]
       │
       ▼
[POST /api/email-tasks]
       │
       ├── EmailStr 校验 ──失败→ 400 返回
       │
       └── 校验通过
              │
              ▼
       [insert_tasks_batch()]
              │
              ▼
       [email_tasks.db: status='pending']
              │
              ├── 调度器扫描 (60s 间隔)
              │       │
              │       ├── send_time <= NOW
              │       │       │
              │       │       ├── 发送成功 → status='sent', sent_at=NOW
              │       │       └── 发送失败 → status='failed', sent_at=NOW
              │       │
              │       └── send_time > NOW → 等待下一次扫描
              │
              └── 手动发送 (POST /api/email-tasks/{id}/send)
                      │
                      ├── 发送成功 → status='sent'
                      └── 发送失败 → status='failed'
```

### 9.2 数据保留策略

| 策略 | 当前状态 | 建议 |
|------|----------|------|
| 已发送记录 | 永久保留 | 可增加定期清理（如保留 90 天） |
| 失败记录 | 永久保留 | 可增加重试次数限制后自动标记终态 |
| 待发送记录 | 直到被调度器处理 | 增加过期时间，超期未发的 pending 任务自动标记 expired |
