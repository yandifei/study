## 1. 引言

### 1.1 编写目的

本文档旨在明确「邮件批量调度系统」(Nebula Mail) 的功能需求与非功能需求，为后续的设计、编码、测试阶段提供依据。

### 1.2 适用范围

本文档适用于：
- 系统架构师：理解功能边界，进行技术设计
- 开发工程师：明确接口契约和业务逻辑
- 测试工程师：制定测试计划和用例
- 项目管理者：跟踪需求实现进度

### 1.3 术语定义

| 术语 | 说明 |
|------|------|
| 渲染模板 | 预定义的 HTML 邮件样式，通过变量占位符替换生成最终邮件 |
| 邮件任务 | 一条待发送/已发送/发送失败的邮件记录 |
| 定时调度器 | 后台线程，周期性扫描到期任务并触发发送 |
| 模板变量 | 渲染模板中的占位符，如 `{user_name}`、`{verification_code}` |

## 2. 项目概述

### 2.1 项目背景

企业和开发者在运营过程中需要向用户发送各类事务邮件（验证码、订单通知、物流更新等）和营销邮件（促销活动、周刊资讯等）。手动逐封发送效率低下，且难以管理定时发送需求。本系统旨在提供一套完整的邮件批量调度解决方案。

### 2.2 项目目标

1. 支持 **11 种邮件场景** 的渲染模板
2. 实现 **批量创建** 邮件任务（多收件人、同一模板）
3. 支持 **定时发送**（可指定未来任意时间）
4. 提供 **Web 可视化管理界面**
5. 内置 **邮箱格式校验**（基于 Pydantic EmailStr）
6. 支持 **TXT 文件批量导入** 收件人

### 2.3 用户角色

| 角色 | 职责 |
|------|------|
| 运营人员 | 通过 Web UI 创建邮件任务、选择模板、设置定时 |
| 系统管理员 | 配置 SMTP 服务、监控发送状态 |

## 3. 功能需求

### 3.1 模块划分

```
┌─────────────────────────────────────────────────────┐
│                  Nebula Mail 系统                      │
├─────────────┬──────────────┬────────────┬───────────┤
│  模板渲染    │  任务调度     │  Web 管理   │  邮箱校验  │
│  (11 种)    │  (定时/即时)  │  (UI + API) │  (Pydantic)│
└─────────────┴──────────────┴────────────┴───────────┘
```

### 3.2 FR-01：邮件模板系统

**优先级**：P0（核心功能）

| 编号 | 模板代码 | 场景 | 关键变量 |
|------|----------|------|----------|
| T01 | `VERIFY_CODE` | 邮箱验证码 | `verification_code` |
| T02 | `PASSWORD_RESET` | 密码重置 | `user_name`, `reset_link`, `expire_hours`, `ip`, `os_browser` |
| T03 | `PASSWORD_CHANGED` | 密码修改通知 | `user_name`, `change_time`, `ip`, `device` |
| T04 | `LOGIN_ALERT` | 异地登录提醒 | `user_name`, `login_time`, `location`, `ip`, `device`, `review_link` |
| T05 | `EMAIL_CHANGE` | 邮箱变更确认 | `user_name`, `old_email`, `new_email`, `cancel_link` |
| T06 | `ORDER_CONFIRM` | 订单确认 | `user_name`, `order_id`, `items[]`, `total`, `pay_method`, `order_url` |
| T07 | `SHIP_NOTIFY` | 物流发货 | `user_name`, `order_id`, `tracking_no`, `tracking_url`, `carrier`, `address` |
| T08 | `REFUND_STATUS` | 退款进度 | `user_name`, `order_id`, `refund_amount`, `status`, `reason` |
| T09 | `ACCOUNT_WELCOME` | 注册欢迎 | `user_name`, `onboarding_steps[]`, `verify_link`(可选) |
| T10 | `PROMO_BATCH` | 促销优惠券 | `user_name`, `event_title`, `coupon_code`, `goods[]`, `banner_url`, `unsubscribe_url` |
| T11 | `NEWSLETTER` | 周刊资讯 | `user_name`, `subject`, `articles[]`, `unsubscribe_url` |

**验收条件**：
- [x] 每个模板均为独立 HTML 文件，包含紫色主题样式
- [x] 每个模板有对应的 Python 发送函数
- [x] 模板变量支持简单类型（str/int）和复合类型（JSON 列表）

### 3.3 FR-02：邮件任务管理

**优先级**：P0

| 功能点 | 描述 |
|--------|------|
| 批量创建 | 一次请求创建多个收件人的同模板任务 |
| 任务查询 | 支持按状态过滤（pending / sent / failed） |
| 任务删除 | 支持删除单个任务 |
| 立即发送 | 忽略定时，即刻发送指定任务 |
| 到期发送 | 手动触发所有到期任务的发送 |
| 统计面板 | 实时展示 total / pending / sent / failed 计数 |

### 3.4 FR-03：定时调度

**优先级**：P1

| 功能点 | 描述 |
|--------|------|
| 启动扫描 | 服务启动时立即检查并发送所有到期任务 |
| 周期扫描 | 后台线程每 **60 秒** 扫描一次 |
| 未来定时 | 支持设定任意未来时间，到期自动发送 |
| 立即发送 | 不填时间则默认为当前时间（即时发送） |

### 3.5 FR-04：邮箱校验

**优先级**：P1

| 功能点 | 描述 |
|--------|------|
| 前端预检 | 提供「校验邮箱」按钮，调用后端 API 逐条校验 |
| 后端强校验 | 创建任务时使用 Pydantic `EmailStr` 强制校验 |
| 行号定位 | 校验结果返回行号，精确定位错误邮箱 |
| TXT 导入 | 支持 `.txt` 文件批量导入（一行一个邮箱） |
| 错误提示 | 前端在校验面板中逐行展示 ✓/✗ 结果 |

### 3.6 FR-05：Web 管理界面

**优先级**：P1

| 功能点 | 描述 |
|--------|------|
| 主题切换 | 支持夜间模式 (dark) 和白天模式 (light) |
| 动态表单 | 选择模板后自动渲染对应的变量输入字段 |
| 拖拽导入 | TXT 文件拖拽/选择导入收件人 |
| 实时统计 | 顶部统计卡片自动刷新（30s 周期） |
| Toast 通知 | 操作结果以 Toast 提示 |

## 4. 非功能需求

### 4.1 性能需求 (NFR-01)

| 指标 | 要求 |
|------|------|
| 单次批量创建 | 支持 100+ 收件人同时创建 |
| API 响应时间 | 非发送类接口 < 200ms |
| 定时扫描间隔 | 可配置，默认 60s |
| 数据库写入 | 线程安全（SQLite WAL 模式 + 锁保护） |

### 4.2 可用性需求 (NFR-02)

| 指标 | 要求 |
|------|------|
| UI 响应式 | 支持 600px ~ 1440px 宽度范围 |
| 浏览器兼容 | Chrome / Edge / Firefox 最新版 |
| 主题持久化 | 主题选择保存到 localStorage |

### 4.3 安全性需求 (NFR-03)

| 要求 | 实现 |
|------|------|
| SMTP 凭据保护 | 通过环境变量注入，不写入代码或配置文件 |
| 邮箱校验 | 基于 RFC 5322 的 Pydantic EmailStr 校验 |
| 输入防注入 | SQLite 参数化查询 |

### 4.4 可维护性需求 (NFR-04)

| 要求 | 实现 |
|------|------|
| 模板扩展 | 新增模板只需添加 HTML + 注册函数到 `DISPATCH_MAP` |
| 字段定义 | 模板变量统一定义在 `TEMPLATE_DEFINITIONS` 字典中 |
| 日志覆盖 | 所有关键操作均有日志记录 |

## 5. 用例描述

### UC-01：创建即时发送任务

| 项目 | 内容 |
|------|------|
| 参与者 | 运营人员 |
| 前置条件 | 已打开 Web 管理界面 |
| 主流程 | 1. 选择邮件模板 → 2. 填写模板变量 → 3. 输入收件人邮箱 → 4. 点击「校验邮箱」→ 5. 确认通过 → 6. 点击「创建任务」→ 7. 后台调度器立即发送 |
| 后置条件 | 任务状态变为 sent 或 failed |
| 异常流程 | 邮箱校验失败 → 显示具体错误行号 → 修正后重试 |

### UC-02：创建定时发送任务

| 项目 | 内容 |
|------|------|
| 参与者 | 运营人员 |
| 前置条件 | 已选择模板和填写变量 |
| 主流程 | 1. 输入收件人 → 2. 打开「定时发送」开关 → 3. 选择未来时间 → 4. 点击「创建任务」→ 5. 任务状态为 pending → 6. 到达设定时间后自动发送 |
| 后置条件 | 任务按计划时间发送 |

### UC-03：TXT 批量导入收件人

| 项目 | 内容 |
|------|------|
| 参与者 | 运营人员 |
| 前置条件 | 准备好 .txt 文件（一行一个邮箱） |
| 主流程 | 1. 点击「从 TXT 文件导入」→ 2. 选择文件 → 3. 系统读取并填充到文本框 → 4. 点击「校验邮箱」→ 5. 确认结果 |
| 后置条件 | 合法邮箱可用于创建任务 |

## 6. 数据字典

### 6.1 邮件任务 (email_tasks)

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | INTEGER | 是 | 自增主键 |
| user_email | TEXT | 是 | 收件人邮箱地址 |
| send_time | TEXT | 是 | 计划发送时间（ISO 8601） |
| template_code | TEXT | 是 | 模板代码（如 VERIFY_CODE） |
| template_vars | TEXT | 是 | 模板变量（JSON 字符串） |
| status | TEXT | 是 | pending / sent / failed |
| created_at | TEXT | 是 | 任务创建时间（ISO 8601） |
| sent_at | TEXT | 否 | 实际发送时间 |

### 6.2 模板变量 (template_vars)

各模板变量定义详见 `utils/email_task_db.py` 中的 `TEMPLATE_DEFINITIONS` 字典，字段属性包括：

- `key`：变量名（对应模板中的 `{变量名}`）
- `label`：中文标签
- `type`：text / number / email / url / json
- `required`：是否必填
- `placeholder`：输入提示
